Codeception was started in November 2011, and released the 1.0 stable version in January 2012. It's a mature and stable project, well-tested and ready to use. It's open-source and MIT licensed.

Popular Questions

Q

Do I really need this?

If you develop a complex web applications with forms, links, and many pages, apis, and you want to check all forms and links in a minutes, you need to use testing framework. Codeception suites the best such tasks.

Q

Why should I use this instead of PHPUnit?

Being the most popular unit testing framework PHPUnit has very limited features for functional testing with Seleniumor other backends. Codeception is PHPUnit on steroids. Everything you need for testing is built-in and works just out of the box. No more pain in configuring data cleanup, paths, Selenium connections, and fixtures. 

Q

No need for this. It looks just like Behat

Hold on! Even it uses the same style in test description and Mink as a driver, Codeception is completely different. As you know, Behat is not a testing framework, but Codeception is. Codeception tests are written in PHP they are written much faster, especially if you use IDE . You can use variables and operators in your tests. You can use CSS and XPathselectors in your tests. This features allows you to build a solid test automation platform for testing your web application. Even though Codeception test are very readable for your developers, managers, and QA team.

We are planning to use Selenium. Why Codeception?

Codeception works great with Selenium. But Codeception provides much more flexible and powerful language then Selenium IDE does. And well, you don't need to learn Java or Python for complex Selenium tests, PHP is ok. And the main reason is: Selenium tests are tightly bound to XPath locators. If you ever change anything in layout tests will fall. Codeception locators are more stable. You can use names, labels, button names and CSS to match elements on page. It's much easier to support the Codeception test then Selenium's one.

Why do all tests starts with 'Guy'?

I'm not a sexist in any form, but we call the test worker a Guy. Think of it as a imaginable person that describers his actions in PHP DSL. I am new guy, he says. I want to test your application, he says. I click "Log In", fill fields, and I see "Hello" in page, he says. In our team we just thought it is more natural to write tests from a person's perspective, then just commands.
